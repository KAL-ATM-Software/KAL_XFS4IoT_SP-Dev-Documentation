    IncludeFileInfoInStacktrace

    If set to false file information will be excluded from stack frames. Defaults to true.

    Declaration
    public static bool IncludeFileInfoInStacktrace { set; }
    Property Value
    Type Description
    bool
    Remarks

    Defaults to true. Normally we want to include file information (including file paths) in the stack trace to make debugging easier, but that makes unit testing more difficult. So for unit tests, where the stack needs to be predicable, set this to false.

    Fail<T>(string)

    Fail immediately

    Declaration
    public static T Fail<T>(string Message = null)
    Parameters
    Type Name Description
    string Message
    Returns
    Type Description
    T
    Type Parameters
    Name Description
    T
    Remarks

    This version gives Fail() a type, which is useful when using inline. For example,

    var x = y!=null ? "Some value" : Fail<string>( "Something failed" );

    It can also be used to indicate to the compiler that Fail() will not return, but 'throwing' an exception. The exception will never actually be thrown because Fail() will never return, but the compiler will assume that anything after Fail() can't be reached - which is correct.

    int fn() 
{
    throw Fail<Exception>( "Failed will never return" ); 
    // unreachable code. 
}

    Just calling Fail() will cause the compiler to complain that fn() doesn't return a value, but throwing a Fail lets the compiler know that the following code can't be called.
